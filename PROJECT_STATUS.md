# ChatInsight — статус проекта

Платформа анализа переписок: backend (**ASP.NET Core .NET 9** + PostgreSQL + Ollama)
и frontend (**React + Vite + Tailwind**). Идея — в `1.docx`.

---

## Сверка с идеей

**MVP 1.0 — закрыт полностью** (импорт, хранение, активность, скорость ответов,
графики, AI-отчёт, PDF). Из **Версии 2.0** сделаны **AI-хронология (Life Timeline Engine)** и **эволюция
личности** (сравнение портретов по периодам — «каким был / стал»).

> `net9.0` вместо `.NET 10`; `pgvector` пока не подключён.

---

## Что работает ✅

### Backend
- [x] Web API (.NET 9), Swagger, PostgreSQL + EF Core, Docker Compose
- [x] Импорт с дозагрузкой (по `SourceId`/`TelegramId`), сброс кэша AI
- [x] 9 модулей аналитики + баланс отношений + сравнение периодов
- [x] AI-инсайты, AI-портреты, **AI-хронология жизни**, **эволюция личности** (Ollama, JSON Schema, кэш в БД)
- [x] **Асинхронный AI**: очередь (`Channel`) + `BackgroundService`, задачи и статусы,
      дедупликация, переживание рестарта
- [x] PDF-отчёт с AI-анализом и портретами

### Frontend
- [x] Загрузка, список чатов, страница анализа
- [x] Графики: по дням, по часам, по авторам; эмоции; баланс отношений
- [x] AI-блоки **асинхронно** (спиннер + опрос статуса): инсайты, портреты,
      **хронология жизни**, **эволюция личности**, сравнение; скачивание PDF

---

## Эндпоинты

| Маршрут | Метод | Что делает |
|---|---|---|
| `/api/import/telegram` | POST | импорт/дозагрузка |
| `/api/chats` · `/{id}` · `/{id}/report` · `.pdf` | GET | список / метаданные / отчёт / PDF |
| `/api/chats/{id}/insights` · `/personality` · `/compare` | GET | синхронные AI/сравнение (кэш) |
| `/api/chats/{id}/insights/async` · `/personality/async` · `/lifetimeline/async` · `/evolution/async` | POST | поставить фоновую AI-задачу → `jobId` |
| `/api/jobs/{jobId}` | GET | статус задачи + результат |

---

## Хранение

PostgreSQL, EF Core. Таблицы: **Chats** (+`SourceId`), **Messages**, **Insights**,
**Personalities**, **LifeTimelines** (кэш хронологии), **AiJobs** (фоновые задачи).

---

## AI-слой

Ollama локально, модель `llama3.1:8b`, structured output по JSON Schema, кэш всех
тяжёлых результатов в БД. Тяжёлые вызовы идут через фоновую очередь; фронт опрашивает
статус. Воркер сериализует результат в camelCase (совпадает с фронтом).
